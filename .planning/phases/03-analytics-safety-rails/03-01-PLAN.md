# Plan 01: Analytics Tab For Nike Vs New Balance Demo

<objective>
Build the Phase 3 analytics tab as a polished, analytics-only campaign dashboard comparing two seeded demo campaigns: `Nike - Alphafly 3` and `New Balance - 530`. The page should communicate spend, revenue, ROAS, budget pacing, and safety impact in the style of the provided ad-platform analytics reference.
</objective>

<context>
Phase 3 is being planned before Phase 2 is implemented. The analytics tab must therefore use deterministic seeded analytics data and should not depend on a live bidding feed, bidding controls, auction UI, or Phase 2 event plumbing. The implementation should make the demo believable now and leave a clean seam for Phase 2 to later replace the seeded data with live simulation-derived analytics.

The visual reference is the user-provided dashboard image: dense ad-platform analytics cards, a large spend/revenue chart, a budget pacing panel, safety guardrail summaries, and lower comparison tables. Ignore the left sidebar from the reference; this work belongs under the existing app's `Analytics` tab/nav item.
</context>

<requirements_addressed>
- METRICS-01: Analytics panel shows spend and remaining budget per campaign/agent.
- METRICS-02: Analytics panel shows impressions, bids won, clicks, and conversions.
- METRICS-03: Analytics panel shows average CPC and ROAS.
- METRICS-04: Analytics panel shows count of skipped unsafe placements.
- METRICS-05: Analytics display updates from app state/data and is structured for live updates later.
- SAFETY-01: Budget cap and budget remaining are visibly represented.
- SAFETY-02: Max CPC guardrail is visibly represented.
- SAFETY-03: Blocked/unsafe prompt counts are visibly represented.
- DASH-03: The analytics tab should show pause/kill-switch state as an analytics/status indicator only if already available; do not build operational pause behavior here unless trivial.
</requirements_addressed>

<truths>
- This is an analytics-only tab. Do not build a live bidding feed, bid buttons, auction winner module, prompt stream, or decision controls.
- The two demo campaigns are exactly:
  - `Nike - Alphafly 3`
  - `New Balance - 530`
- The demo prompt context is running-performance demand around: `how can i get to my goal of a 4 minute kilometer`.
- The analytics tab should show what happened after spend occurred: spend, revenue, ROAS, conversions, budget remaining, efficiency, and safety impact.
- Since Phase 2 is absent, use seeded deterministic data that looks like a plausible 14-day campaign run.
- The page should look like a professional ad-spend platform analytics view, using the current Next.js/Tailwind/lucide stack and the existing clean AgentBid Studio style.
- Keep the implementation scoped to the analytics tab and reusable analytics data/types. Do not rewrite the Phase 1 agent creation flow.
</truths>
